# ADR 005: 시간 한도는 단계별 컴포넌트 하나가 책임진다

- Status: Accepted
- Date: 2026-05-03
- 관련: 000, 001, 004

## 문제

한 요청 안에 여러 타이머가 동시에 살아 있으면, *어느 한도가 실패를 만들었는지*가 흐려진다. 동시에 살아 있는 타이머가 많을수록, 취소 처리 순서에 따라 버그가 생길 수 있는 경우의 수가 크게 늘어난다.

## 결정

**요청의 생애주기를 시간축으로 자르고, 각 구간의 한도를 한 컴포넌트가 책임진다.**

- **요청 전체 시간(wall-clock)** → Handler. 호출 진입에서 한 번 건다. 모든 단계가 이 한도 안에 들어간다. 호출자에게 노출되는 상한.
- **비스트림에서 한 시도 시간** → routing.Service. 시도가 끝나면 즉시 풀린다. 폴백을 트리거하는 자연스러운 짝이라 [ADR 004](004-fallback-policy.md)와 같은 컴포넌트에 둔다.
- **스트리밍 중 이벤트 사이 유휴(idle) 시간** → streamRelay. 누적 한도가 아니라 사이 간격 — 길게 흐르는 응답이 자연스럽게 통과한다.

### 경계선

- **스트리밍 첫 이벤트 도착 한도는 따로 두지 않는다.** 어댑터는 호출자의 ctx만 따른다. 별도 타이머는 취소 순서를 다루는 비용에 비해 이득이 적다.
- **회로 차단의 쿨다운 / 백오프는 *간격*이지 *한도*가 아니다.** [ADR 004](004-fallback-policy.md)의 영역.
- **운영자는 전체 한도를 *첫 이벤트가 도착하기까지의 상한*으로 본다** — 전체 한도를 길게 잡으면 첫 이벤트가 오기 전에도 그만큼 기다릴 수 있다.

## 근거

- 동시에 살아 있는 타이머가 줄어, 취소 순서 버그가 생길 여지가 줄어든다.
- `record.error_kind`가 컴포넌트의 책임으로 바로 매핑된다 (Handler / Service / streamRelay).
- 새 한도를 더할 때도 같은 잣대를 쓴다 — *어느 단계의 권위자에 둘 것인가*부터 묻는다 ([ADR 001](001-component-boundaries.md)).

## 결과

- 운영자가 세 한도(전체 / 시도당 / 유휴)를 머리에 들고 있어야 한다. 권위자가 컴포넌트에 묶여 있어 추적은 짧다.
