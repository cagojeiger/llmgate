# ADR 004: 폴백 적격성과 회로 차단 — chain 진행 정책

- Status: Accepted
- Date: 2026-05-03

## 배경

ADR 001 이 *별명 + 시도 순서 목록* 으로 폴백 가능성을 열었지만, *언제 다음 후보로 넘어갈지* 와 *반복 실패 모델을 어떻게 잠재울 것인가* 는 결정으로 남았다. 모든 에러를 폴백 사유로 두면, 잘못된 키처럼 *모든 후보가 같은 이유로* 실패하는 케이스가 chain 끝까지 가서야 fail 한다 (호출자 입장에선 fail 까지 시간이 chain 길이만큼 늘어남, 디버깅에선 사실은 키 문제인데 모델 문제로 보임). 반대로 실패가 누적된 모델을 무한히 두드리면 다른 호출도 같은 지연을 진다. 두 측면 — *어떤 에러가 폴백 적격인가* 와 *반복 실패 모델을 어떻게 다룰 것인가* — 는 모두 *chain 진행을 멈추는 결정* 이라 한 ADR 에 묶었다.

## 결정

- **폴백 적격성은 에러 종류로 판정** — request 자체가 잘못된 케이스 (bad_request) 와 인증 실패 (auth) 는 *어느 vendor 로 보내도 같은 결과* 일 가능성이 높다. 일시 실패 (rate_limit, upstream 5xx, timeout, network) 만 chain 을 진행시킨다.
- **auth 는 디폴트 적격에서 제외** — 키 mis-config 의 swap-mask 위험. 모든 후보가 같은 키 실수로 401 이 나면 두 번 fail = 사실은 키 문제인데 모델 문제로 보인다. 운영자가 의도적으로 켜야만 폴백된다.
- **운영자가 디폴트 변경 가능** — 폴백 적격 에러 셋은 env 한 개로 override. 환경별로 다른 정책을 yaml 에 박지 않는다 (ADR 002 의 *정책은 env*).
- **회로 차단은 cooldown + 지수 백오프 + 지터** — 한 모델이 연속 실패 N 회 하면 차단. 차단 동안 후보에서 자동 skip. 차단 시간이 지나면 *모두에게 다시 열린다*. 반복 차단되면 차단 시간이 늘어나고 (백오프), 같은 시점에 같은 모델로 모두가 한꺼번에 들어가지 않게 시간에 ±지터를 입힌다. 상한 (cap) 을 두어 영원히 잠재워지지 않게 한다.
- **명시 half-open probe 단계는 두지 않는다** — 차단 끝의 첫 호출이 사실상 probe 역할이고, 명시 probe 단계는 (a) 어떤 호출이 probe 가 되는지의 간헐적 latency 비결정성, (b) probe 동안의 추가 분기, (c) "왜 이 호출만 늦는가" 의 운영 mental model 비용 — 셋 다 *한 머리에 들어옴* 원칙 위반. 호출량이 큰 환경에선 어차피 자연 평탄, 호출량이 적으면 가끔 *운 나쁜* 1 회가 발생하지만 해당 호출은 자체 폴백 chain 안에서 다음 후보로 자연 진행한다.
- **회로 상태는 프로세스 메모리** — 외부 store 로 인스턴스 간 공유하지 않는다 (ADR 001 *상태 없음*). 다중 인스턴스 환경에선 인스턴스마다 회로를 따로 들고 있는 비효율을 받아들인다.

## 결과

좋아지는 점:
- 운영자 mental model 단순. record 의 attempts 만 보면 *왜 이 chain 이 어디서 끝났는가* 가 즉시 보인다.
- 영구 실패 (bad_request / 디폴트 auth) 는 첫 후보에서 즉시 종료 → 호출자가 빠르게 본다. 일시 실패만 chain 을 끝까지 시도한다.
- 회로 동작이 *cooldown 한 개* 로 설명된다. probe 단계라는 *준-차단* 상태가 없어서 모델 상태가 둘 (열림 / 닫힘) 뿐이다.

받아들이는 단점:
- auth 디폴트 제외는 운영자가 *swap-mask 위험을 의식해서* 의도적으로 auth 를 켤 때만 폴백된다 — 디폴트가 안전 쪽으로 기울어 있다.
- half-open 미도입은 차단 끝 직후 첫 호출이 다시 차단으로 가는 *운 나쁜 1 회* 가능. 호출량이 적은 환경에서 종종 보일 수 있다 — 받아들임.
- 회로 상태가 인스턴스 별이라, 같은 모델이 인스턴스 A 에서 차단 중이어도 인스턴스 B 에선 다시 호출된다. 다중 인스턴스에서 vendor 부담이 인스턴스 수에 비례.

## 다른 선택지

- **모든 에러를 폴백 적격으로** — bad_request / auth 의 swap-mask 가 chain 끝까지 끌고 간다. 호출자 latency 만 늘고 디버깅이 흐려진다. 폐기.
- **probe-driven half-open** — 위 (a)·(b)·(c) 비용. 폐기.
- **모델 별 / 별명 별 정책 (별명마다 다른 적격 셋, 다른 백오프)** — 그 신호가 V1 에서 없음. 필요해지면 별명 yaml 안에 정책 override 를 추가하는 식으로 확장 (ADR 002 가 같은 자세).
- **회로 상태를 외부 store 로 공유** — 인스턴스 간 회로 일관성. ADR 001 *상태 없음* 위반. 다중 인스턴스 비효율을 받아들이는 쪽이 정체성에 맞다.
