# ADR 003: 의도적 미지원 — 안 하는 것 모음

- Status: Accepted
- Date: 2026-05-03

## 배경

ADR 001 이 게이트웨이의 정체성을 *상태 없음 / 사실만 / OpenAI 모양 / 별명 + chain* 으로 박았다. 그 정체성으로부터 따라오는 *무엇을 하지 않는가* 가 줄줄이 흩어져 있었다. 흩어두면 같은 외부 요청 (`/v1/models 추가` / 사용량 한도 검사 / hot-reload / 모델 가격표 / 키 별 쿼터 분배 등) 이 들어올 때마다 같은 토론을 다시 한다. 모아두면 같은 잣대로 빠르게 거절할 수 있고, 다른 ADR 들이 *왜 이 결정을 했는가* 의 frame 으로 003 을 참조하게 된다.

## 결정

V1 에서 다음을 *지원하지 않는다*. 각 항목은 ADR 001 의 어느 축이 거절 근거인가를 함께 적는다.

- **mid-stream 폴백** — 첫 이벤트가 클라이언트에 송출된 뒤의 vendor 교체. HTTP 시맨틱 (200 OK 이후 partial body 가 이미 갔는데 응답 substitute 불가), SDK 호환 (호출자는 누적 메시지를 한 모델 답으로 가정), record 무결성 (모델 X·Y 의 토큰이 한 record 안에 섞이면 평가·단가·디버깅 모두 부서짐) — 셋 다 ADR 001 의 *사실만* 축을 위반. 경계의 디테일은 ADR 006.
- **capability matching / `/v1/models` discovery** — 게이트웨이가 모델의 능력 / 가능한 호출 모양을 *판정* 하면 파생 값을 만드는 것. ADR 001 의 *파생 값을 직접 만들지 않는다* 위반. 능력은 호출자가 안다.
- **hot-reload** — yaml 갱신을 실시간 반영. 운영자 시점에서 *언제 반영됐는가* 가 흐려지고, 같은 호출 안에서 부분 적용된 catalog 를 보는 race 가 생긴다. 재시작 = 적용 (ADR 001 의 *상태 없음* 과도 자연 정렬).
- **모델 메타정보 (가격 / context window / capabilities) 보유** — yaml 에 박지 않는다. destination 없음 (위의 capability matching · /v1/models 둘 다 안 함). 가격은 후처리 시스템이 record 를 받아 계산한다. ADR 002 가 같은 결론을 catalog 면에서 박았다.
- **multi-key smart distribution** — 한 모델에 여러 API 키를 두고 round-robin / weighted / quota-aware 로 분배. 키별 사용량 상태가 필요 → ADR 001 *상태 없음* 위반. 운영자가 같은 vendor model 을 다른 yaml 두 장에 다른 인증 환경으로 등록하면 catalog 안에서 별개 모델이 되고, alias chain 의 자연 단위로 들어간다 (ADR 002 의 부수 효과). smart 분배가 필요해지면 *후처리 시스템* 이 chain 순서를 바꾸는 식으로 표현 가능.
- **사전 한도 차단 (pre-call quota)** — 호출 전에 한도 검사. 외부 상태 조회 = 상태 없음 깨짐. 한도는 후처리 시스템이 record 를 모아 판정한다.
- **k8s / CRD 인지 / 게이트웨이 안의 operator 패턴** — 게이트웨이가 k8s 를 알면 다른 환경 (compose / bare metal / 로컬 dev) 운영 부담이 늘고 게이트웨이의 정체성에 *환경 의존* 이 새로 들어온다. ADR 002 결정대로 게이트웨이는 yaml 디렉토리 1 개를 받을 뿐, mount 가 어디서 오는지 모른다. operator 가 필요해지면 *외부* 컴포넌트가 yaml 을 굴린다 — 게이트웨이는 mount path 만 본다.

## 결과

좋아지는 점:
- 외부 요청에 같은 잣대로 응답 가능 — "이건 003-항목 X 에 해당해서 안 합니다". 운영 면적이 V1 가설대로 작게 유지된다.
- *경계가 분명한 게이트웨이* 는 다음 시스템 (operator / 후처리 / dashboard) 을 깨끗하게 받는다 — 그쪽이 얹혀 가야 할 책임이 흐려지지 않는다.
- 다른 ADR 들이 003 을 frame 으로 참조한다 — 같은 근거 (정체성) 가 매 ADR 에서 반복되지 않는다.

받아들이는 단점:
- 비싼 prompt 의 사전 차단 / capability mis-match 의 친절한 에러 / 운영자 GUI hot-reload — 모두 게이트웨이 *밖* 에서 해결해야 한다.
- "이 정도면 가벼우니 게이트웨이가 해도 되지 않나" 의 누적 압력. 항목이 늘면 ADR 003 갱신으로 응답 — 디폴트는 *거절*.

## 다른 선택지

- **결정마다 따로 ADR 두기** — 같은 근거 (ADR 001 정체성) 가 매 ADR 에서 반복되고, *전체* 무엇이 빠져 있는지 한눈에 안 들어온다. 폐기.
- **architecture.md 의 한 절로만** — 살아 있는 reference 에는 적합하지만 *시점 박힌 결정 근거* 가 약하다. 살아있는 명세는 architecture.md, 결정 근거는 003 — 두 문서가 다른 깊이로 같이 산다.
