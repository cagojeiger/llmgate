# ADR 004: 폴백은 일시적 실패와 송출 전 스트림에만 허용한다

- Status: Accepted
- Date: 2026-05-03
- 관련: 000, 001, 005

## 문제

별명 체인으로 폴백을 열어 두기는 했지만, 세 가지가 결정되지 않았다 — 어떤 오류에서 다음 후보로 넘어갈 것인가, 반복 실패하는 모델을 어떻게 잠시 제외할 것인가, 스트리밍 중 어디까지 폴백을 허용할 것인가.

모든 오류를 폴백 적격으로 두면 키 설정 실수가 체인 끝까지 따라간다. 스트리밍 폴백의 경계가 흐릿하면 응답 도중에 모델이 바뀌어 감사 기록 무결성과 SDK 호환이 같이 깨진다.

## 결정

**일시적 실패만 다음 후보로 넘긴다.** `rate_limit`, 5xx 같은 `upstream`, `timeout`, `network` 네 가지에서만 폴백한다. `bad_request`나 `auth`는 즉시 종결한다 — 영구적 실패는 첫 후보에서 빠르게 끊어, 호출자가 빠르게 받는다.

`auth`는 디폴트로 *제외*한다 — 키 설정 실수가 다른 모델로 옮겨붙는(swap-mask) 위험이 있다. 운영자가 바꾸려면 `LLMGATE_FALLBACK_ON` env로 켠다 ([ADR 002](002-catalog-shape.md)).

**반복 실패한 모델은 회로 차단으로 잠시 제외한다.** 연속 N번 실패하면 모델을 차단하고, 차단 동안엔 후보에서 건너뛴다. 쿨다운 + 지수 백오프 + 지터 + 상한(cap)으로 동작 — 모델 상태는 두 가지뿐(열림 / 닫힘). 회로 상태는 프로세스 메모리에만 두고, 인스턴스끼리 공유하지 않는다 ([ADR 000](000-identity.md)).

**스트리밍 폴백은 송출이 시작되기 전까지만 허용한다.** 응답 상태(status open)와 첫 이벤트(first event) 단계는 어댑터가 검증하고, 적격이면 폴백한다. 첫 청크가 클라이언트로 나간 뒤(mid-stream)에는 벤더를 교체하지 않는다.

### 경계선

mid-stream 폴백을 거부하는 이유 셋:

- **HTTP 의미** — 200 OK + 부분 응답이 이미 나갔다.
- **SDK 호환** — 호출자는 한 모델의 답으로 가정한다.
- **감사 기록 무결성** — 모델 X와 Y의 토큰을 이어붙이면 평가·과금·디버깅이 모두 깨진다.

토큰 한 개라도 송출됐으면 다른 모델로 대체할 수 없다 — 의도된 결과. 스트리밍 도중 실패는 SSE 에러 이벤트 + 종결 신호로 처리하고, 이미 받은 청크는 그대로 둔다.

half-open 탐침(probe) 단계도 *두지 않는다* — 차단이 풀린 직후의 첫 호출이 자연스러운 탐침 역할을 한다.

## 근거

- 영구적 실패는 첫 후보에서 즉시 끊어, 호출자가 빠르게 실패를 받게 한다. 일시적 실패만 체인 끝까지 시도한다.
- 회로 차단 동작을 쿨다운 하나로 이해할 수 있다.
- 첫 이벤트 검증이 어댑터의 영역이라, 새 벤더를 추가할 때도 검증 로직이 어댑터 안에 머문다 — Service는 고칠 필요가 없다 ([ADR 001](001-component-boundaries.md)).

## 결과

- `auth`를 제외하는 디폴트라 운영자가 의도적으로 켜야 한다. 디폴트가 안전 쪽.
- half-open을 두지 않으니, 차단이 풀린 직후의 첫 호출이 *운 나쁜 한 번*으로 다시 차단될 수 있다.
- 회로 상태를 인스턴스별로 두기 때문에, 다중 인스턴스에서는 벤더가 받는 부담이 인스턴스 수에 비례한다.
- 첫 이벤트 직후에 끊기면 호출자에게 거의 빈 응답이 간다.
