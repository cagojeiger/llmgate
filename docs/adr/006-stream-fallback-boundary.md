# ADR 006: 스트리밍 폴백 경계 — 첫 이벤트가 마지막 갈림길

- Status: Accepted
- Date: 2026-05-03

## 배경

스트리밍은 *한번 시작되면 되돌릴 수 없는* 응답이다. 어디까지가 "아직 클라이언트에 약속을 안 했나" 를 명확히 그어야 폴백이 안전하다. 경계가 흐리면 mid-stream 폴백이 슬그머니 들어와 record 무결성 / SDK 호환 / HTTP 시맨틱을 동시에 깬다. ADR 003 의 *mid-stream 폴백 미지원* 결정을, 이 ADR 에서 *언제 폴백이 가능한가* 의 양면에서 확정한다.

## 결정

스트리밍 한 호출을 시간축으로 세 단계로 본다 — (1) **status open**: upstream 이 스트림 시작에 응답한 시점 (HTTP status 검증), (2) **first event**: 첫 이벤트 도착 검증, (3) **mid-stream**: 그 이후. (1)·(2) 는 폴백 가능 영역, (3) 은 폴백 *불가* 영역.

- **첫 이벤트 도착이 폴백 가능 마지막 경계** — 클라이언트에 200 OK + 첫 chunk 가 송출되기 *전* 까지가 chain 진행 가능. 그 후의 실패는 다른 vendor 로 갈아타지 않는다.
- **검증은 어댑터 안** — 벤더 와이어 차이 (sentinel / 정상 종료 모양 / 에러 envelope) 는 어댑터 내부 도메인이다. 라우터는 검증된 결과만 받는다. 라우터에 와이어 도메인이 새어 들어가지 않게 한다 (ADR 007 의 컴포넌트 책임 경계).
- **status open 단계 실패도 같은 폴백 적격 규칙으로** — 상태 코드를 어댑터가 분류해 일반 호출 에러로 모양을 맞춘다. 라우터는 스트림이든 non-stream 이든 같은 *적격 / 회로* 판정 (ADR 004) 을 적용. 스트리밍을 위한 별도 폴백 분기를 두지 않는다.
- **mid-stream 실패는 그 응답 안에서 종결** — 클라이언트에 SSE 에러 이벤트 + 종결 신호로 *왜 끊겼는지* 만 통지. 받은 chunk 는 그대로 남는다. 다른 vendor 로 갈아타기 시도 안 함.
- **mid-stream 폴백 거부 근거 셋**:
  - HTTP 시맨틱: 200 OK + partial body 가 이미 송출됨. 응답 substitute 불가.
  - SDK 호환: 호출자는 누적 메시지를 한 모델의 답으로 가정. 모델 X·Y 토큰을 봉합하면 *다른 모델 응답을 하나처럼* 보이게 만드는 것이고, finish_reason / usage / model id 가 record 와 어긋난다.
  - record 무결성: 한 record 안에 두 모델 토큰이 섞이면 평가·단가·디버깅 모두 부서진다 — ADR 001 *사실만* 위반.

## 결과

좋아지는 점:
- 라우터가 mid-stream 케이스를 *볼 일이 없다* — 라우터 책임이 chain 진행으로 좁아진다.
- 스트리밍과 non-streaming 의 폴백 의미가 같은 단어 (chain / 적격) 로 설명된다. 다른 점은 *폴백 가능 시점이 어디까지인가* 뿐.
- 첫 이벤트 검증이 어댑터 도메인이므로, 새 vendor 추가 시 검증 로직도 어댑터 내부에서 끝난다. 라우터는 변경 영역 밖.

받아들이는 단점:
- 첫 이벤트가 도착한 *직후* connection 이 끊기면 클라이언트는 거의 빈 응답을 받는다. 폴백으로 구할 수 있을 것처럼 보이지만, 1 토큰이 이미 송출되어 substitute 불가 — 의도된 결과.
- 첫 이벤트 도착 전까지의 latency 는 클라이언트 입장에서 *모델이 답하는 속도* + *vendor 연결 시간* 으로 한꺼번에 보인다. 폴백이 일어나면 그 시간이 chain 길이만큼 누적 (단, 회로 차단이 영구 실패한 모델을 자동 skip 해서 누적 상한이 자연 잡힘).

## 다른 선택지

- **mid-stream 폴백 시도** — HTTP / SDK / record 셋을 동시에 위반. 폐기.
- **first-event 검증을 라우터로 이동** — 어댑터의 와이어 도메인이 라우터로 새어들어가 라우터가 비대해진다. ADR 007 컴포넌트 책임 경계 위반. 폐기.
- **status open 만으로 폴백 결정 (첫 이벤트 검증 생략)** — 일부 vendor 가 200 OK 만 던지고 첫 이벤트 직전에 close 하는 케이스를 못 잡음 → status open 통과한 뒤 첫 chunk 가 *영원히 안 옴*. 첫 이벤트까지 봐야 안전. 폐기.
- **mid-stream 에서 partial output 을 폐기하고 폴백** — 호출자에 보낸 byte 를 회수할 수 없다. HTTP 시맨틱상 substitute 자체가 정의되지 않음. 폐기.
