# ADR 007: 컴포넌트 책임 경계 — 한 머리에 들어오는 단위로 자른다

- Status: Accepted
- Date: 2026-05-03

## 배경

디스패처 한 곳에 *체인 진행 + 폴백 적격 + 회로 + 첫 이벤트 검증 + 시간 한도 분기 + record 채우기* 가 모여 있던 시기엔 "스트리밍이 왜 이렇게 복잡한가" 를 추적할 때 디스패처부터 봐야 했고, 다른 부분의 변경이 디스패처까지 닿았다. ADR 001 의 *한 사람 머리에 들어오는 작은 컴포넌트* 가 코드 단위에선 깨져 있었다. 책임을 *동시에 한 머리에 들어와야 할 단위* 로 다시 자른다.

## 결정

각 컴포넌트는 *유일한 책임* 을 가진다. 다른 ADR 들이 결정한 사항이 자연 자리에 가도록 책임을 묶는다.

- **handler** — 요청을 받고 / 디스패처로 보내고 / non-stream 과 stream 분기를 결정. 요청 총 wall-clock 한도의 권위자 (ADR 005). 호출자에 노출되는 HTTP 시맨틱의 경계.
- **gateway router** — 별명 chain 을 풀고 / 적격 에러는 다음 후보 / 회로 차단 상태를 본다 (ADR 004). non-stream 시도당 한도의 권위자 (ADR 005).
- **adapter** — 한 vendor 의 와이어 도메인. HTTP 호출, status 분류, 와이어→공통 모양 정규화. 스트리밍에서는 첫 이벤트 검증까지 (ADR 006).
- **stream 응답기** — 스트림이 열린 *이후* 의 SSE 와이어 transcript. idle / cancel / 에러 매핑 / 종결. 스트림 idle 한도의 권위자 (ADR 005).
- **audit recorder** — 요청당 사실 1 개 발행 (ADR 001). 호출자 / 디스패처 / 응답기가 채운 record 를 sink 로 흘려보낸다.

책임 경계 원칙:

- **한 결정엔 권위자 1 개** — 같은 결정에 두 컴포넌트가 같이 발언하지 않는다 (시간 한도는 단계당 한 컴포넌트, 폴백 적격성은 gateway router 만, mid-stream 종결 모양은 응답기만, 와이어 검증은 adapter 만).
- **다른 컴포넌트의 도메인을 모른다** — 디스패처는 SSE 와이어를 모른다. 응답기는 별명 / 폴백 / 회로를 모른다. 어댑터는 chain 을 모른다. 한 컴포넌트의 변경이 다른 컴포넌트의 invariant 를 모르는 채 통과해선 안 된다.
- **분리의 동기는 *책임이 둘* 이라는 신호** — 컴포넌트가 커진다고 자르지 않는다. 두 책임이 한 컴포넌트 안에 같이 살고 있어서 한 책임의 변경이 다른 책임의 invariant 를 깨기 쉬울 때만 자른다. 신호가 없으면 *작아도* 합치지 않고, *커도* 가르지 않는다.
- **추상 1 개를 더 두지 않는다** — pipeline / middleware / interceptor 같은 추상은 *그 추상의 invariant* 를 또 한 컴포넌트가 책임지게 만든다 (chain 순서 / signature / 에러 흐름). 추상이 늘면 mental model 비용이 컴포넌트 분리 이상으로 증가한다. 우리 규모에선 직접 컴포넌트 분리가 우월.

## 결과

좋아지는 점:
- 변경 영역 추적이 컴포넌트 단위로 끝난다 — idle 동작 변경은 응답기만, 폴백 적격성 추가는 디스패처만, 새 vendor 추가는 어댑터 1 개.
- 기여자가 *한 컴포넌트만 머리에 담고* 작업 가능. ADR 001 정신을 코드 단위에서 회복.
- 같은 책임이 *두 컴포넌트에 새어 들어가는 신호* 가 빠르게 보인다 — 변경 PR 이 두 컴포넌트를 같이 건드리면 책임 경계를 의심하는 trigger.
- 다른 ADR 들이 *결정을 어느 컴포넌트의 책임으로 둘 것인가* 를 본 ADR 의 권위자 표를 보고 정한다 — 결정 자리가 흐려지지 않는다.

받아들이는 단점:
- 단일 호출이 컴포넌트를 여럿 거친다. 단일 호출 trace 의 *수직 길이* 가 늘어난다 — 책임이 명확한 trace 가 *짧지만 모호한* trace 보다 디버깅 친화. 받아들임.
- 작은 변경도 책임 경계를 한번 묻게 된다 — *맞는 컴포넌트에 두는가* 의 review 비용. 비용이 invariant 깨짐 비용보다 낮다.

## 다른 선택지

- **단일 god-component** — 출발점. 책임이 한 컴포넌트에 누적되어 변경 영역이 모든 책임에 닿음. 폐기.
- **pipeline / middleware 추상** — 위 *추상 1 개를 더 두지 않는다* 항목. 우리 규모엔 mental model 비용 우위. V1 미적용.
- **기능 단위 분리 (스트리밍 / non-stream / 폴백 / 회로 각각의 모듈)** — 한 호출이 *모든* 모듈을 거치므로 책임 분리가 아닌 *유형 분류* 에 가까워진다. 본 ADR 의 *동시에 한 머리* 원칙과 충돌. 폐기.
- **컴포넌트 더 잘게 쪼개기 (예: 디스패처를 chain-진행 / 폴백 / 회로 셋으로 분리)** — 책임이 *한 결정에 같이 발언* 하는 셋이라 분리가 invariant 를 흩는다. 폐기.
