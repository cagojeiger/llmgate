# ADR 005: 타임아웃 권위자 분리 — 한 단계당 한 권위자

- Status: Accepted
- Date: 2026-05-03

## 배경

한 요청에 시간 한도가 *겹쳐서* 작용하면 (요청 총 시간 + 시도당 시간 + 첫 이벤트까지 + 이벤트 사이 + ...) record 만 보고 *어느 한도가 어느 fail 을 만들었는지* 가 흐려진다. 같은 호출에 동시에 살아있는 timer 가 많을수록 *같은 동작이 동시에 두 timer 를 cancel 시키는 race* 표면이 곱연산으로 늘어난다. 이전엔 스트림 시작 한도 + 요청 총 한도 + 시도 한도가 한 라우터 안에 같이 살았고, "성공했지만 timer 가 한 박자 늦게 fire 한 race" 를 우회하기 위한 곡예 (성공 신호와 timer 신호 중 누가 먼저인지 별도 분기) 가 들어와 있었다. 시간 권위자가 컴포넌트마다 깨끗이 한 명이 되도록 한도를 책임자별로 자른다.

## 결정

요청 생애주기를 시간축으로 자르고, 각 구간에 *시간 한도의 권위자* 를 한 컴포넌트로 둔다.

- **요청 총 wall-clock 한도 — handler 가 권위자**. 호출 진입에서 1 회 설정. 아래 모든 단계가 이 한도 *안에* 산다. 호출자에 노출되는 *상한*.
- **non-stream 시도당 한도 — router 가 권위자**. 한 모델 호출이 너무 길면 다음 후보로 넘기기 위함. 시도가 끝나면 즉시 해제. 폴백 트리거의 자연 반차원이라 chain 진행과 같은 컴포넌트에 둔다 (ADR 004).
- **스트림 idle 한도 — 응답기가 권위자**. 첫 이벤트 이후, 다음 이벤트가 한도 동안 안 오면 종결. *누적* 한도가 아니라 *이벤트 사이* 한도이므로 긴 응답이 자연 통과한다.
- **스트림 시작 / 첫 이벤트 별도 한도는 두지 않는다** — 첫 이벤트 도착은 *요청 총 한도* 안에서 어댑터가 호출 컨텍스트만 의지해 처리한다. 별도 timer 를 두면 "성공했지만 timer 가 늦게 fire 한 race" 를 우회하기 위해 *권위자가 둘 이상* 인 곡예가 들어온다 (ADR 006 의 첫 이벤트 검증 책임이 어댑터에 사는 것과 정확히 짝).
- **회로 차단 시간 / 백오프 / 지터 — 시간 *간격* 이지 시간 *한도* 가 아니다**. 한 호출의 fail 을 만들지 않으므로 본 ADR 의 영역 밖, ADR 004 의 회로 책임.

## 결과

좋아지는 점:
- 한 호출에 동시 살아있는 timer 가 줄어 race 표면이 작다.
- record 의 error_kind 가 어느 권위자에서 나왔는지 *컴포넌트 책임* 으로 즉시 매핑된다 (handler-총 / router-시도 / 응답기-idle). 디버깅이 짧아진다.
- 한도 추가 요청에 같은 잣대 — *어느 단계의 권위자에 새 한도를 두어야 하는가* 부터 묻는다. 새 권위자를 만드는 변경은 ADR 갱신.

받아들이는 단점:
- 운영자가 3 개의 한도 (총 / 시도 / idle) 를 머리에 들어야 한다 — 그래도 권위자가 컴포넌트로 묶여 있어 *어디서 어떻게 fail 했나* 가 짧게 추적된다.
- 첫 이벤트 도착 한도가 별도로 없으므로, 요청 총 한도가 길게 잡혀 있으면 첫 이벤트가 나오기 전에도 그 한도까지 기다린다. 운영자는 총 한도 자체를 *첫 이벤트 기대 한도의 commit* 으로 본다.

## 다른 선택지

- **모든 단계에 별 timer (per-attempt + per-first-event + per-idle + ...)** — race 표면이 곱연산. 컴포넌트 간 cancel 신호가 엮여 시간 흐름 추적 비용이 늘고, 한 단계의 변경이 다른 단계에 invariant 부담을 만든다. 폐기.
- **단일 총 한도만** — 시도당 폴백 트리거가 사라진다 (한 모델이 총 한도까지 끌면 다음 후보 시도 자체를 못 함). 폐기.
- **첫 이벤트 별도 한도 부활** — race 곡예 비용 + 권위자 둘. 운영 mental model 에서 *총 한도* 를 첫 이벤트 commit 으로 보면 별 한도 불필요. 이전 시도를 양보 결정으로 폐기.
