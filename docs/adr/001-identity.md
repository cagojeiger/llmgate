# ADR 001: 정체성 — 상태 없는, 사실만 흘려보내는 게이트웨이

- Status: Accepted
- Date: 2026-05-02

## 배경

agent 마다 LLM 호출 / 폴백 / 키 관리 / 토큰 계산을 다시 짜고 있었고, 운영 중 모델 사용 / 폴백 사유에 대한 시야가 없었다. 기존 풀-피처 게이트웨이는 운영 면적이 커서, 한 사람 머리에 들어오는 작은 컴포넌트 (~2K 줄) 가 필요했다.

## 결정

**상태를 저장하지 않는, OpenAI SDK 가 기대하는 모양의 게이트웨이. 사실만 발행하고 파생 값은 직접 만들지 않는다.**

- **상태 저장 안 함** — DB / 캐시 / 공유 저장소 없음. 회로 차단 같은 휘발 정보만 프로세스 메모리에 산다.
- **OpenAI SDK 모양** — 호출자는 OpenAI SDK 를 그대로. 다른 벤더는 어댑터에서 OpenAI 모양으로 정규화.
- **별명 + 시도 순서 목록** — 카탈로그 별명이 모델 시도 순서로 풀린다. 첫 후보 실패 시 다음.
- **사실만 발행** — 요청당 audit Record 한 줄을 stdout 로. 비용 / 한도 / 집계는 후처리 시스템 책임.
- **카탈로그는 외부 파일** — `catalog/` 디렉토리 (모델 / 별명 yaml). hot-reload 없음, 바꾸려면 재시작. 자세한 모양은 ADR 002.

## 결과

좋아지는 점:
- 운영 단순. 잃을 영구 데이터 없음.
- 후처리 도구 자유 (게이트웨이는 사실 발행자 역할만).
- Record 하나로 폴백 이력 추적.

받아들이는 단점:
- 사전 한도 차단 불가 (외부 상태 조회 = 상태 없음 깨짐).
- 회로 차단이 프로세스마다 따로.
- 후처리 시스템 없으면 비용 안 보임.
- 캐시 없음.

## 다른 선택지 검토

- **풀-피처 게이트웨이** — 운영 면적이 우리 문제 대비 과대. 비용 / UI / 정책 엔진은 영역 밖.
- **agent 안에 그대로 두기** — agent 간 동작 어긋남, 운영 시야 없음. 떼어내는 게 동기.
- **상태 있는 게이트웨이** — 사전 한도 / 캐시 가능, 대신 운영 부담. V1 단순함 가설과 충돌.
