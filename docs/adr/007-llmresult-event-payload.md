# ADR 007: 결과 이벤트 본문 정책은 consumer 단위 capture 모드로 정의한다

- Status: Accepted
- Date: 2026-05-26
- 관련: 003, 005

## 문제

결과 이벤트의 request·response 본문에는 운영자가 통제할 수 없는 데이터가 들어간다.
컨슈머가 보낸 프롬프트와 모델이 만든 응답은 PII, 자격증명, 영업 비밀, 규제 대상
데이터를 자연스럽게 포함할 수 있고, 한 번 브로커 보관소에 들어가면 그 보관소를
다루는 모든 사람이 본다.

본문을 일률적으로 끄는 답도 안전망이 아니다. 어떤 컨슈머는 메타데이터·usage만으로
충분하지만, 다른 컨슈머는 재현·재처리·품질 분석을 위해 본문이 반드시 필요하다. 본문
발행 여부는 게이트웨이 차원에서 일괄 결정할 일이 아니라 각 컨슈머의 정책이다.

정책이 없으면 본문이 늘 흐른다. 그 결정이 어디에서도 내려지지 않았기 때문에 발행
경로의 다른 결정들 — 누적 상한, truncate 규칙, 페이로드 크기 — 도 임의로 결정되고,
결과적으로 브로커의 페이로드 상한을 넘기는 순간 손실이 일어난다. 본문 정책이 먼저
서야 그 위의 결정들이 일관되게 따라온다.

## 결정

**본문 발행은 consumer 단위 `body_capture` 모드로 정의한다. 기본은 `omit`이며,
컨슈머가 명시적으로 옵션을 선언해야 본문이 발행된다.**

- `consumers/<name>.yaml`에 신규 필드:
  ```yaml
  body_capture: omit | plain   # 기본: omit
  ```
- 모드별 event payload 형태:

  | 모드 | request / response | 추가 필드 |
  | --- | --- | --- |
  | `omit` | `null` | `body_capture: "omit"` |
  | `plain` | 원본 그대로 | `body_capture: "plain"` |

- 본문을 담는 모드(`plain`)는 per-field byte cap을 적용하고, 초과 시
  `request_truncated` / `response_truncated` flag를 세운다. 스트리밍 누적도 같은 cap
  안에서 멈춘다.
- 다운스트림 컨슈머는 항상 `body_capture` 필드를 먼저 보고 본문 존재 여부를 결정한다.

### 경계선

- **메시지 dedup 키 정책은 이 ADR의 범위 밖.** 클라이언트 제어 `X-Request-Id`가
  그대로 브로커 dedup 키로 쓰이는 문제는 별도 변경에서 server-generated ID로 분리한다.
- **per-model·per-request override는 도입하지 않는다.** 정책 표면이 늘어나 감사하기
  어려워진다. 모델별로 본문 정책이 다른 경우라면 그 모델을 부르는 컨슈머를 별도로
  등록한다.
- **vendor 4xx error body·attempts.Error는 정책 대상이 아니다.** 메타데이터로 분류한다.

## 근거

- **본문 발행 결정의 단위는 컨슈머다.** 본문은 컨슈머가 만들어 넣는 데이터이고,
  그 데이터의 민감도와 활용처를 아는 사람도 컨슈머 운영자다 — [ADR 003](003-consumers.md)이
  정한 "컨슈머 = 데이터 정책의 단위" 원칙의 자연스러운 연장.
- **기본은 닫혀 있어야 한다.** 컨슈머가 명시하지 않은 데이터 노출은 사고다. ADR 003이
  consumers 디렉토리 부재시 부팅 fail로 잡은 것과 같은 형태로, 본문 발행도 명시적
  선언이 있어야 일어난다.
- **본문 정책은 그 위의 모든 결정을 떠받치는 1차 결정이다.** 누적 상한, truncate
  마커, 페이로드 크기, 발행 채널 분리, 본문 보호 수단 같은 후속 결정은 본문 발행
  여부가 정해진 다음에야 일관되게 정할 수 있다. 그래서 코드 변경이 아닌 정책 ADR로
  먼저 못 박아 둔다.

## 결과

- `consumers/<name>.yaml`에 `body_capture` 필드를 둔다. 부재 시 `omit`. 본문이 필요한
  컨슈머는 yaml에 명시적으로 `plain`을 선언한다.
- `Event` schema에 `body_capture` 필드를 둔다. 다운스트림은 이 값으로 본문 존재 여부를
  분기한다.
- 누적 상한·truncate 마커·dedup 키 분리·본문 보호 수단 같은 후속 결정은 별도 PR /
  스펙으로 이어진다. 이 ADR은 그 결정들이 어떤 정책 표면 위에서 동작할지를 고정한다.
